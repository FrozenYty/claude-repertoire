# CLAUDE.md 指南

[English version](README.md)

一份构建和使用 `CLAUDE.md` 的实用指南 —— 它是塑造 Claude Code 在项目中工作方式的行为约束文件。

## 什么是 CLAUDE.md？

`CLAUDE.md` 是一个放在项目根目录的纯 Markdown 文件。Claude Code 在每个会话开始时会读取它，并遵循其中的指令。可以把它理解为：

- **行为 linter** —— 它在常见 LLM 错误发生之前就能捕获它们（过度设计、投机性功能、跨文件漂移）。
- **共享的心智模型** —— 你和 Claude 在一个地方一次性达成规范共识，而不是每次会话都重复同样的纠正。
- **项目上手文档** —— 新成员（人类或 AI）阅读它就能知道如何操作。

它并不是伪装成 `prompt` 的东西。它是一组常设指令 —— 适用于每项任务的规则，而不是针对某项具体任务的指令。

## 为什么做这个

大多数 CLAUDE.md 文件要么是空的，要么只有一句话，要么是项目构建命令的堆砌。很少有人把它当作设计对象来对待。

本指南的观点是：一份精心编写的 CLAUDE.md 是你在 Claude Code 项目中杠杆率最高的投入。本仓库中的九条规则提炼自真实会话 —— 每条规则的缺失都曾导致实际的 bug、浪费了时间，或产生了不可维护的输出。每条规则在 `examples/` 和 `principles/` 中都有具体的案例研究作为支撑。

## 快速开始

1. 阅读 [principles/00-three-tier-system-zh.md](principles/00-three-tier-system-zh.md)
   了解 CLAUDE.md 文件存放位置及其交互方式。
2. 将 `CLAUDE.md` 模板复制到用户级（`~/.claude/CLAUDE.md`）存放
   通用规则，或复制到项目根目录存放项目专属规则。
3. 删除不适用于你工作流程的规则。
4. 添加一条项目专属的规则（例如你的技术栈、命名规范）。
5. 使用一周。留意哪些地方仍然出错。为它加一条规则。
6. 重复。一份好的 CLAUDE.md 永远不会完成，只会趋于收敛。

## 九条规则速览

| # | 规则 | 一句话总结 |
|---|------|---------------------|
| 1 | 编码前先思考 | 声明假设。展示权衡。先问后猜。 |
| 2 | 简单优先 | 解决问题所需的最少代码。无投机性功能。 |
| 3 | 外科手术式修改 | 只触碰必须改的地方。匹配现有风格。 |
| 4 | 目标驱动执行 | 定义可验证的成功标准。循环直至通过。 |
| 5 | 语言规则 | 对话用中文，文件写英文。全角标点。验证 Unicode。 |
| 6 | 输出工作区 | 工作区根目录 `CLAUDE_CODE_FILES/`，支持工作区级覆盖。按日期创建子文件夹。 |
| 7 | 交叉引用纪律 | 审计每次变更的波及范围。修复所有过时引用。 |
| 8 | 生成物自检 | 每个产出物附带结构化检查清单。 |
| 9 | 子 Agent 调度 | 并行独立工作。向无上下文 Agent 提供简要说明。验证输出。 |

## 设计哲学

三条原则塑造了本文件的每条规则：

1. **预防胜于纠正。** 一条能在错误发生前阻止它的规则，抵得上事后十次“请修正这个”的纠正。本文件中的每条规则都追溯到某类特定的 bug，这些 bug 在规则被写下之前反复出现。

2. **可验证，而非空泛。** “写出好代码”毫无用处。“每个生成的产出物附带一份包含是/否检查项的结构化清单”是可证伪的。每条规则都在陈述要做什么（DO），而非推崇什么（VALUE）。

3. **最小表面积。** 规则不是教程。它们短小、命令式、密度高。如果一条规则需要超过三个要点来解释，就把它拆成两条规则，或者将解释移入 `principles/`。

## 贡献

这是一份活的文档。如果你发现了能持续改善 Claude Code 会话的规则，欢迎提交 issue 或 PR。

- 新增规则需要：一个名称、一句话的粗体总结、2-4 个要点，以及在 `examples/` 目录下提供一个该规则本可预防 bug 的真实案例研究。
- 修改规则需要：一份展示前后对比的更新版案例研究。
