# 案例研究：Unicode 引号审计

[English version](unicode-quote-audit.md)

## 涉及哪些规则

- **规则 5（语言规则）** —— 中文排版子规则和 Unicode 验证子规则。
- **规则 8（生成物自检）** —— 结构化检查清单本可在交付前捕获这些问题。

## 发生了什么

在对 `papersmith` 技能 24 个 prompt 文件进行全面审计期间，一个扫描 Unicode 引号错误的 Python 脚本在 `humanize-zh.md` 中发现了五处不同的 bug —— 而这个文件正是*为整个工具包定义中文排版规则的文件*。

这五处 bug 分为三类：

1. **中文上下文中的 ASCII 引号**：第 4 行使用了 `"machine flavor"` 和 `"translationese"`，使用了 ASCII `"`（U+0022）—— 直接违反了工具包自身的中文排版规则，因为其所在段落并非完全为英文。

2. **引号方向颠倒**：第 14 行将 `"de-AI"` 包裹为 `U+201D...U+201C`（右引号在前，左引号在后）。引号是全角的，但方向是反的 —— 这是人类读者无法察觉但在字节层面可检测的 bug。

3. **左引号用作右引号**：第 18 行以 `"`（U+201C，左引号）而非 `"`（U+201D，右引号）结束一个示例短语，造成该行左引号与右引号比例为 4:2。

## 哪里出了错

三个叠加的失效：

1. **视觉不可区分**：在大多数等宽编辑器中，`""`（U+201C/U+201D）和 `""`（U+0022）渲染效果相同。人类校对者浏览文件时看不出差别。

2. **工具层面的归一化**：`Edit` 工具默默拒绝了将 ASCII 引号替换为全角引号的尝试，报告"无差异" —— 因为对这些工具的比较逻辑而言，这些字符看起来是一样的。

3. **灯下黑**：那个教工具包如何使用正确中文引号的文件，本身就是仓库中引号 bug 最多的文件。没有人对规则文件运行过验证脚本。

## 这些规则如何预防它

规则 5 的 Unicode 验证子规则写道："用 `python -c 'print([hex(ord(c)) for c in line if ord(c)>127])'` 来验证。"在 `humanize-zh.md` 上运行这个单行命令，本可在十秒内捕获全部五处 bug。

规则 8 会要求一个自检项，例如"所有紧邻中文文本的 `"` 均为 U+201C/U+201D，而非 U+0022" —— 一个可证伪的是/否检查。

## 关键启示

**你无法用肉眼审计 Unicode。** 当正确性依赖于区分 `""` 与 `""` 或 `"` 与 `"` 时，请使用十六进制级别的脚本。相信你的眼睛，正是一份排版规则文件带着五处引号 bug 发布的原因。
