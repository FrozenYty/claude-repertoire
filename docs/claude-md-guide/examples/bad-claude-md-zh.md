# 糟糕的 CLAUDE.md：新手反模式

[English version](bad-claude-md.md)

这是一个开发者第一次尝试写 CLAUDE.md 时的典型产物 —— 在理解什么能让行为约束真正生效之前。每一行要么太模糊无法执行，要么太模糊无法验证，要么根本就不是一条常设规则。

---

## 新手版本

```
# CLAUDE.md

Write good code.
Don't make mistakes.
Use best practices.
Follow the project conventions.
Be helpful and professional.
今天帮我写登录页面。
Write tests for everything.
Keep the code clean.
Always use the latest version of dependencies.
Make it fast.
```

十行。每一行都有问题。

---

## 逐行分析

### ❌ “Write good code”

**为什么失败：** “好“没有可操作的定义。两个开发者（更不用说两个 LLM 会话）每次都会有不同的解读。它无法被验证，也无法被执行。

**对应规则 4（目标驱动执行）。** 一条规则必须陈述*如何*达成目标，而不只是表达愿望。正确形式：”为每项任务定义可验证的成功标准。循环直至通过。”

**同时对应规则 8（生成物自检）。** “好“通不过可证伪性测试。正确形式：”每个产出物附带一份包含是/否检查项的结构化清单。'看着不错'不算一项。”

---

### ❌ “Don't make mistakes”

**为什么失败：** 它命名了负面结果（“错误”），但完全没有提供预防机制。这等同于告诉一个司机“别撞车”，却不提后视镜、转向灯或限速。

**对应规则 1（编码前先思考）。** 真正能预防错误的规则是：“明确声明你的假设。如果不确定，就问。如果有不清楚的地方，停下来，指出困惑所在。”

---

### ❌ ”Use best practices“

**为什么失败：** “最佳实践”是一个修辞上的捷径，实际意思是“我今天想到的任何东西”。它传递不出任何具体信息。更糟的是，它会诱使过度设计 —— 没人要的抽象、设计模式和可配置性。

**对应规则 2（简单优先）。** 正确的指令是：“解决问题所需的最少代码。不添加未要求的功能。不为一次性代码引入抽象。”这才具体、才可证伪。

---

### ❌ “Follow the project conventions”

**为什么失败：** 这预设了规范的存在，却没有指出任何一条。如果规范在另一个文件中，就链接到它。如果没有写下来，就无可遵循。这条指令浪费了一行空间，却没有增加任何约束。

**对应规则 3（外科手术式修改）。** 一条有用的规范规则会明确说明要匹配什么：“匹配现有的缩进、命名和导入风格。只触碰必须改的地方。每一行修改都应能直接追溯到用户的请求。”

---

### ❌ ”Be helpful and professional“

**为什么失败：** 这是一个价值观陈述，不是一条指令。LLM 无法执行”表现得专业” —— 它需要具体的行为：语气规则、回复结构、何时提出异议、何时提出澄清性问题。

**对应规则 1（编码前先思考）。** 对应的行为等价物是：“如果存在更简单的方法，就说出来。在必要的时候提出异议。如果存在多种解读，展示它们 —— 不要默默选择一种。”

---

### ❌ “今天帮我写登录页面”

**为什么失败：** 这是一条任务 prompt，不是一条常设规则。CLAUDE.md 在每个会话开始时都会被读取。像”今天写登录页面“这样的具体任务，到第二天就毫无意义，到第三天就令人困惑，而当它导致 Claude 重新执行一个已经完成的任务时，就变得有危害性了。

**对应设计哲学（README）。** CLAUDE.md 是”一组常设指令 —— 适用于每项任务的规则，而不是针对某项具体任务的指令“。任务属于聊天 prompt。规则属于 CLAUDE.md。永远不要混淆二者。

---

### ❌ ”Write tests for everything“

**为什么失败：** “一切”没有边界，也不切实际。Claude 应该为一行配置改动写测试吗？应该为 README 编辑写测试吗？这把一个合理的目标变成了噪音。不指定*哪种*测试，LLM 可能会写出表面通过、实则什么也捕获不到的测试。

**对应规则 4（目标驱动执行）。** 正确的指令将测试与具体成功标准挂钩：“写一个能复现 bug 的测试，然后让它通过。”测试是验证的工具，而非目的本身。

---

### ❌ “Keep the code clean”

**为什么失败：** 和 “Write good code” 是同一个模式。“干净”是主观的、不可验证的。一个人眼中的干净，是另一个人眼中的过度抽象。这个词带有情感色彩，但精确度为零。

**对应规则 2（简单优先）。** 替换为：“如果你写了 200 行而 50 行就能解决，重写它。一个高级工程师会认为这过度复杂吗？”这给出了一个具体的启发式方法，而非一个模糊的形容词。

---

### ❌ “Always use the latest version of dependencies”

**为什么失败：** 盲目升级依赖会导致问题。这条规则自相矛盾 —— 第二行说“不要犯错”，第九行说“总要升级一切”。一条常设规则不该强制要求破坏性操作。

**对应规则 7（交叉引用纪律）。** 当依赖变化时，你必须检查所有引用它们的文件：”每次变更都有波及范围。在将任务标记为完成之前，审计所有受影响的文件。”正确的行为是*审计波及范围*，而不是*总是升级*。

---

### ❌ “Make it fast”

**为什么失败：** 没有目标。没有指标。无法知道何时完成。“快”对于 CLI 工具可能意味着 10ms，对于 web 请求可能意味着 100ms —— LLM 无从选择。

**对应规则 4（目标驱动执行）。** 性能和其他成功标准一样。正确的形式要提供目标：“该端点必须在 p95 下 200ms 内响应。”如果你无法量化目标，就无法验证它。

---

## 结构性问题

除了行与行之间的问题之外，这份新手 CLAUDE.md 缺乏架构：

- **没有分组。** 十行平铺直叙。没有理由非得是十条而非一条或五十条。结构良好的 CLAUDE.md 将相关规则归入编号小节，以便 LLM 可预测地浏览。
- **没有优先级。** “Write good code“和”Don't make mistakes“意思大致相同，重复了两次。当一切都同等重要时，就没有什么是重要的。本仓库的九条规则是有序的：规则 1（行动前思考）必须在规则 4（验证结果）之前。
- **没有权衡确认。** 新手文件呈现了一种不可能的理想 —— 所有规则，永远适用。真正的 CLAUDE.md 开篇就是”这些指南偏向谨慎而非速度。对于琐碎任务，自行判断。“这一句话就防止了 LLM 对一个单词的拼写修正运用外科手术式修改规则。

---

## 新手文件做对了什么（偶然地）

- 它存在。一份糟糕的 CLAUDE.md 比没有 CLAUDE.md 好得多得多，因为它给了你改进的起点。这份分析中的每一行都来自一个真实开发者所写的真实文件，他们意识到它有问题，然后替换了它。
- 它很短。十行。保持简洁的本能是对的 —— 本仓库的九条规则正好占满一屏。错误在于用愿望而非指令填充这十行。

---

## 关键启示

你的 CLAUDE.md 中的每一行都必须回答三个问题。如果一行不能全部回答，就删除它或重写它，直到它能够。第一：**Claude 应该执行什么具体行为？** 不是”be helpful” —— 那是价值观。“在写代码之前明确声明你的假设”才是行为。第二：**你如何验证合规性？** 不是”write good code“ —— 你检查不了。”每个产出物附带一份包含是/否检查项的结构化清单“是可以检查的。第三：**这适用于每一项任务，还是一次性指令？** 如果是一次性指令（”写登录页面“），它不属于 CLAUDE.md。如果是常设约束（”每次编辑都匹配现有风格“），它就属于。新手 CLAUDE.md 和专业 CLAUDE.md 的差距不在于长度、格式或巧妙程度，而在于每一行是否通过这个三问测试。本页开头那个版本有十行，零行通过。去检查你项目中的 CLAUDE.md，对照这个测试。你大概率会发现至少有一行需要重写。
