# 4. 目标驱动执行

[English version](04-goal-driven-execution.md)

## 为什么要有这条规则

模糊的成功标准会产生两种对称的失败：agent 停得太早（交出一个“看起来做好了”但经不起细看的东西），或者永远不停（因为没有“够好”的定义而无休止地打磨）。两者都源于同一个根因——任务从未被表述为一个可证伪的条件。

“让输出看起来专业”不是一个成功标准。它无法被验证。agent 无法围绕它循环，因为“专业”没有二元答案。结果要么是一个带着格式错误过早交付，要么是一个无休止的边距调整循环。

一个具体案例：中文排版自检。标准不是“确保引号看起来正确”——无法验证。而是“中文字符相邻处零个 ASCII 双引号字符（U+0022）”。可证伪。每次编辑后，运行`check_quotes.py`看到`0 issues`就是一个二元的通过/失败。agent 可以独立循环：编辑、验证、编辑、验证——直到脚本返回零。不需要人的判断。

## 实践中的样子

**模糊：** 任务：“重构auth模块。”成功标准：“代码应该更干净。”agent 重组了函数、重命名了变量、宣布胜利。三周后：token 刷新坏了——但“更干净”达到了，所以 agent 认为任务完成。

**可证伪：** 任务：“重构auth模块。”计划：(1) 将token逻辑提取到`token.rs`中——验证：`cargo test auth::`通过。(2) 用`AuthError`枚举替换临时的错误字符串——验证：`cargo build`零警告。(3) 移除遗留函数——验证：`grep -r legacy_ src/`返回空。每一步都有一个二元检查。

## 何时可以放宽

探索性任务（“调查为什么登录页很慢”）一开始缺乏二元标准。交付物是一份报告，不是一个通过的测试套件。将标准从“通过了吗？”转换为“我产出了一个具体的、可执行的发现了吗？”——“可能出在数据库”不算完成；“`GET /session`耗时要2.3秒，因为每次页面加载都执行了`bcrypt.verify`”才算。
