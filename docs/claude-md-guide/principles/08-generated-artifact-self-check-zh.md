# 8. 生成制品自查

[English version](08-generated-artifact-self-check.md)

## 为什么有这条规则

生成 XML、代码或 JSON 的 agent 会自信地交付“看起来没问题”的输出，但在结构审查下就会出问题——未闭合的标签、缺失的必需属性、错误的方向流向、未转义的字符。这些不是隐蔽的 bug；它们十秒钟就能检查出来。但如果没有显式的检查清单，agent 的默认行为是扫一眼输出就宣布完成。

一个具体案例：drawio 图表生成器在写入任何 XML 之前，使用一个包含 13 项的自查清单。第 12 项：“每条前向边满足 `source.y > target.y`。”没有这一项，方向颠倒的堆叠——最常见的 drawio bug——每次都会通过视觉审查。这个检查项让一个不可见的坐标属性变得可见。

同样，绘图代码生成器使用一个包含 10 项的检查清单。第 3 项：“是否设置了 `pdf.fonttype = 42`？”这一行代码就决定了论文是通过 ACM/IEEE 投稿还是在 PDF 检查阶段被拒稿。验证只需一秒——但前提是它出现在清单上。

## 实践中是什么样子

**坏——“看起来没问题”：**
> Agent：“这是你的图表。看起来是正确的。”交付了一个 XML 文件，其中有两个重叠的形状、一条指向已删除节点的边，以及一个与 IEEE 配色方案冲突的硬编码颜色。这三个 bug 本该被检查清单捕获。

**好——结构化验证：**
```
自查：
1. XML 是否格式良好？通过
2. 所有边是否引用了存在的节点？通过
3. 所有前向边是否满足 source.y > target.y？通过
4. 没有超出页面的元素？通过
...
13. 没有重复转义的 &amp;amp;？通过
```
每项是一个是/否问题。用户可以看到哪些项目经过了验证。

## 什么时候可以放宽

对于一次性制品（一次性的数据导出、调试日志），检查清单并不必要。对于用户会直接使用的任何东西——配置文件、生成的脚本、图表、eval——检查清单是不可省略的。
