# 9. 子 Agent 调度

[English version](09-sub-agent-dispatch.md)

## 为什么有这条规则

单个 agent 按顺序处理大型任务会触及上下文上限、在子任务间丧失焦点，并产生不一致的输出。将同一个任务划分给多个子 agent——每个子 agent 有聚焦的范围和明确的交付物——可以更快完成且质量更高。但如果划分不当（范围重叠、上下文不足），会产生重复或冲突的编辑，比顺序执行更糟糕。

一个具体案例：审计 33 个文件，涵盖 prompt、引用、示例和元文档。单个 agent 需要 45 分钟以上，而且很可能遗漏跨文件的不一致之处。取而代之的是，三个 agent 并行运行：一个处理 prompt（22 个文件），一个处理引用 + 示例（10 个文件），一个处理元文档（4 个文件）。每个都在 5 分钟以内提交了报告。关键的是，三者独立地在*不同的文件*中发现了相同的章节编号漂移 bug——提供了汇聚的证据，这是单 agent 顺序审计所遗漏的。

另一个案例：三个 agent 在后台同时重新生成示例（`transformer.drawio`、`diffusion.drawio`、`sota-comparison.py`），而主会话继续处理其他工作。

## 实践中是什么样子

**调度前先问自己：**"这个任务的各个部分可以并行执行吗？"如果可以，就派发出去。顺序逐个执行会触及上下文上限，并在子任务间失去焦点。

**有效调度的关键原则：**

1. **沿自然边界划分**——文件目录、任务类别、互相独立的输出。绝不要让两个 agent 有对同一个文件的写权限。

2. **像对同事一样做简报**——agent 是冷启动的，对你对话中的上下文一无所知。一个好的 prompt 应说明：做什么、为什么重要、文件在哪里、以什么格式报告，以及回复长度上限。

3. **独立任务用后台，依赖任务用前台**——当不需要等待结果就能继续下一步时，使用 `run_in_background`；当需要等待时，使用同步方式。

4. **信任但要验证**——agent 的摘要描述的是它*打算*做什么。在汇报工作完成之前，先阅读实际的改动。

**坏的调度：**
> “审查这个 skill 并修复任何问题。”——范围模糊，没有输出格式，agent 可能修改用户尚未批准的文件。

**好的调度：**
> "读取 `prompts/` 中的所有 22 个文件。对每个文件检查：（A）是否存在 Self-Audit，（B）中文文本附近不得有 ASCII `"`，（C）占位符命名是否一致。按 A/B/C 分组报告，附带行引用。仅报告——不要编辑任何文件。"

## 什么时候可以放宽

对于涉及少于三个文件或一个明确单一操作的任务，子 agent 会增加协调开销而无益。当任务可以被自然地划分，且每个部分足够大以支撑一个完整 agent 上下文时——粗略地说，涉及 5 个以上文件或一个独立的创作性任务——这条规则激活。
