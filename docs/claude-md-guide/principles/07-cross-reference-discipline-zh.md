# 7. 交叉引用规范

[English version](07-cross-reference-discipline.md)

## 为什么有这条规则

LLM 将每项任务视为一个点的修改：编辑一个文件，宣告成功，继续下一项。但软件项目是引用的图。当你重命名文件、移动函数、更新配置值、新增章节或删除组件时 —— 每个指向旧位置的文件现在都是一个 bug。LLM 的本能是在完成主要编辑后就停下来。这条规则迫使它在宣告完成之前执行连带编辑。

数字、章节编号、文件名和配置值是任何项目中被复制粘贴最多的字符串。它们出现在索引表、README、CHANGELOG、eval 描述、路由表以及内联注释中。当你修改源值时，每个副本都会悄悄地漂移 —— 而且这些是代码审查中最难注意到的 bug，因为文本单独看完全合理。错误的章节编号下面不会出现红色波浪线。

一个具体案例：`drawio-templates.md` 被重新组织了 —— 删除了四个模板，新增了十一个。实际的模板移到了新的章节。但文件顶部的索引表从未更新。**十个中有十个**索引条目指向了错误的章节。用户按照“流程图 → §3”操作，会跳转到 RAG 管线模板。

这个错误随后传播到了 `examples/README.md`、`evals/evals.json` 和 `CHANGELOG.md`。三个独立的审计 agent 发现了同一 bug 的不同碎片 —— 没有一个意识到这是系统性的。

另一个案例：开发者在 `config.py` 中将配置字段 `max_retries` 重命名为 `retry_limit`。LLM 更新了定义，但在七个文件中留下了 14 处 `max_retries` 的引用 —— 测试夹具、CLI 参数解析器、环境变量映射器、README 示例。全部 14 处变成了运行时错误。修复只需 30 秒的 grep，但 agent 从未执行。

## 实践中是什么样子

**每次有跨文件影响的修改之后，审计波及范围：**

```bash
grep -r "旧文件名" .          # 文件重命名
grep -r "旧函数名" .          # 函数移动/重命名
grep -r "旧配置键" .          # 配置变更
grep -r "§5" .               # 章节编号变更
grep -r "19 templates" .     # 计数变更
```

在同一个 commit 中修复每一个匹配项。如果旧值在多个文件中是错误的，那这是同一个 bug，而不是 N 个独立的 commit。

## 什么时候可以放宽

如果被修改的值只存在于一个文件中，不会出现在其他任何地方（例如单个脚本中的内部变量名），那么搜索步骤只是走过场。这条规则在某个值已经被文档化、被索引、被其他文件引用、或作为接口暴露时激活 —— 对于文件名、函数签名、配置键、计数和章节编号来说，几乎总是如此。
