# 3. 精准改动

[English version](03-surgical-changes.md)

## 为什么要有这条规则

LLM 编辑文件的方式就像一位好心的客人帮你整理公寓：他们把本来放得好好的东西重新摆放。被要求改一行代码时，agent 常常会“优化”旁边几行的格式、重写一条它觉得不够清晰的注释，或者重构同一个文件里的辅助函数——而这些全都不是被要求做的。这些顺手清理累积起来，会让 diff 充满噪音，真正的改动被埋没在一堆修饰性编辑下面。

在 code review 中，这是灾难性的。审查者打开一个 diff，期望看到一行逻辑修复，结果面对的是三个文件里改了40行——修复本身，加上重新格式化的空白、重命名的变量、外加一个“清理过”的 import 块。审查者只有两个糟糕的选择：花时间从噪声中分离信号，或者信任这个 diff 从而遗漏 bug。

一个清晰的例子：在编辑一个 SKILL.md 的 frontmatter 块以更新 `description` 字段时，agent 完全没有动其他字段——路由表、工具定义、版本字符串——原封不动。diff 恰好只有一行。审查者一眼就能看到。

## 实践中的样子

**坏——范围蔓延：** 被要求改一个 CSS 颜色，agent 还把属性按字母排序、把相邻规则里的`px`转成`rem`、删掉一段被注释的代码因为“这是死代码”。diff 是60行。哪一部分是 bug 修复？一眼望去没人知道。

**好——最小改动：** agent 只改了`background: #f00`这一行为`background: #0f0`，其他一概不动。一行的 diff。如果它注意到死代码，会在回复里说——“顺便一提，第45-48行看起来很可能是不可达代码”——但不去碰它们。

## 何时可以放宽

如果你的改动引入了一个新函数，而这个新函数依赖于一个结构很差的已有 API，且按原样使用该 API 会让你的新代码更糟糕，那么重构那个 API 在改动范围之内——它现在是你引入的依赖，不是你不巧碰到的别人代码。你要对自己写的代码负责，包括它的集成点。但对于仅仅是你滚动经过的代码，你没有打磨的责任。
