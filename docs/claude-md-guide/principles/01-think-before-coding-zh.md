# 1. 先想后写

[English version](01-think-before-coding.md)

## 为什么要有这条规则

LLM 天生倾向于输出内容，而不是停下来承认无知。面对歧义时，LLM 的默认行为是默默猜测——选择统计上最可能的解释然后一路狂奔。这造成了一种危险的失效模式：agent 生成的代码看似合理，却建立在一个错误的假设之上。

一个具体案例：在一次 skill 审计中，某个 agent 被要求生成一个 drawio XML 图。它没有先检查引用的模板是否存在，而是假设了模板路径并据此生成了图。结果引用了一个不存在的文件——从一开始就无效。一个两秒钟的存在性检查就能发现这个问题。

反模式：填补空白是语言模型最擅长的事。但在软件工作中，用猜测来填补空白与引入 bug 是难以区分的。

## 实践中的样子

**坏——默默假设：**
> 用户：“把导出按钮加到toolbar上。”
> Agent：立即写代码，假设右上角toolbar、CSV格式、`download.svg`图标、无确认对话框。三个假设错了两个。

**好——暴露不确定性：**
> 用户：“把导出按钮加到toolbar上。”
> Agent：“两个需要确认的问题：(1) 哪个toolbar——顶部导航还是每个条目旁的？(2) 什么格式——CSV、JSON还是两者都要？确认后大约5行代码。”

## 何时可以放宽

如果任务真的没有歧义——“把`src/parser.rs`里的`tmp`重命名为`buffer`”——跳过提问。这条规则针对的是隐藏的歧义，而非形式主义。一个判断启发式：如果你能在不做任何主观选择的情况下完成实现，直接动手即可。
