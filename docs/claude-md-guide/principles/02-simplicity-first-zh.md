# 2. 简洁优先

[English version](02-simplicity-first.md)

## 为什么要有这条规则

LLM 有一种过度工程化的本能。面对一个简单任务，它会去搞配置标志、插件架构和扩展点——不是因为问题需要这些，而是因为训练数据里充斥着生产级代码。模型会把“解决 X 的代码”模式匹配到最复杂的版本，而不是最简单的那个。

这种倾向的代价尤其高昂，因为 LLM 生成的抽象往往是错误的抽象。它们预判了一些永远不会出现的扩展方向，增加了间接性却没有增加价值。一个单函数功能被包装在一个带策略模式的类里不是更干净——而是更难阅读、更难修改、更难删除。

一个直接案例：`polish-abstract`提示词。需求是“润色这个学术摘要”。agent 可以加一个期刊参数、一个字数限制标志、一套按学科分类的术语模块。但最终它保持了一个扁平的5段结构，零额外参数。没有可以配置错的东西。类似地，写作反模式参考文档只收录了12条——最高频、最致命的错误——而不是追求穷尽。

## 实践中的样子

**过度工程化：** 用户要一个把`.jpeg`改成`.jpg`的脚本。agent 写了120行，包括`--dry-run`、`--recursive`、`--preserve-timestamps`、一个配置解析器和一个日志模块。用户现在得读120行才能信任一个10行代码的操作。

**简洁：**
```python
import os, sys
folder = sys.argv[1]
for f in os.listdir(folder):
    if f.endswith('.jpeg'):
        os.rename(os.path.join(folder, f),
                  os.path.join(folder, f[:-5] + '.jpg'))
```
八行。没有标志。没有抽象。五秒就能读完。

## 何时可以放宽

如果要构建的基础设施会被三个或更多不同的地方调用，做一些前期抽象是合理的。如果用户明确要求可配置性，就加上——但仅限于被要求的那部分。
